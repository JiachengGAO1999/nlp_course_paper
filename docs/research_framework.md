# Research Framework

Role: concise shareable research narrative for collaborators. This file explains
the motivation, core question, design sketch, and current findings. `GUIDE.md`
remains the project command document.

## Project

Working title:

> Keep Recent or Keep Relevant? Position-Dependent Trade-offs in Prompt-Based
> Compression of Multi-Turn Reasoning Histories

This project studies recency-biased prompt-based self-compression for
evidence-bearing multi-turn reasoning histories.

The core research question is:

> Under a fixed compressed-history budget, does retaining recent turns verbatim
> improve compressed multi-turn reasoning histories, or does it create a
> position-dependent trade-off between preserving recent evidence and losing
> older evidence?

The study is about compression architecture as a diagnostic intervention, not
prompt engineering. The compressed conditions use the same compression prompt
template. What changes is how the fixed history budget is allocated: one global
summary versus an older-history summary plus recent verbatim context.

## Motivation

Production LLM systems often compress accumulated dialogue or agent history to
fit a fixed context budget. Two common prompt-based architectures are:

- one-shot full-history summarization,
- summary of older history plus recent turns kept verbatim.

These architectures are common in agent compaction and conversation memory. The
project does not claim that production systems use only these two forms; rather,
it uses them as controlled baselines for a widely used recency heuristic. Recent
turns are often treated as a proxy for relevance. In multi-turn reasoning,
however, answer-critical evidence is not always recent. This creates a possible
relevance-recency mismatch: hybrid memory may protect recent evidence while
starving older evidence of summary budget.

## Data Construction

Layer 1 uses MuSiQue-Ans as the main controlled benchmark.

Each item starts from:

- a benchmark question,
- a gold answer,
- 2-4 supporting evidence paragraphs,
- a reasoning-hop decomposition.

The open-answer QA item is converted into multiple choice:

- gold answer becomes one option,
- three LLM-generated distractors become the other options,
- MC options are automatically audited and manually spot-checked.

The final question and options are held out from the dialogue history. They are
shown only at the final answer stage. The benchmark reasoning decomposition is
also held out: dialogue turns must not expose MuSiQue subquestion or
short-answer labels.

## Dialogue Histories

For each benchmark item, the pipeline generates a multi-turn source-note
collection dialogue.

User turns are scripted to place benchmark evidence and distractors at controlled
positions:

- `far_early`: answer-critical evidence appears early,
- `far_middle`: evidence appears in the middle,
- `cross_turn`: evidence is spread across early, middle, and late turns,
- `late`: evidence appears near the end.

Assistant turns are generated by Qwen3-8B using a natural intermediate reasoning
style. The assistant may interpret visible notes, connect them tentatively to
earlier notes, and discuss uncertainty. The assistant does not see the held-out
final question or answer options and must not use a final-answer format,
subquestion labels, source short-answer labels, or red-herring labels.

This creates realistic mixed user-assistant histories while preserving control
over where the answer-critical evidence appears.
The pipeline also records whether answer-critical evidence appears in the
recent-turn window preserved verbatim by the hybrid condition, so recency effects
can be analyzed rather than silently confounded with architecture effects.

## Conditions

The experiment compares three history conditions:

| Condition | Input to answer model |
| --- | --- |
| `full_history` | Complete uncompressed dialogue history |
| `one_shot_summary` | Full dialogue summarized in one compression call |
| `hybrid_summary_recent` | Older turns summarized, recent turns kept verbatim |

The two compressed conditions use the same compression prompt template and the
same compressed-history budget. The architecture comparison tests whether
recent-turn retention is a generally better memory strategy or a recency-biased
budget allocation with position-dependent costs:

- if answer-critical evidence is in the recent window, hybrid can preserve it
  verbatim while one-shot may omit or blur it during global compression;
- if answer-critical evidence is older, hybrid must preserve it through a
  smaller older-history summary and may regress relative to one-shot;
- if recent turns contain salient distractors, hybrid can retain those
  distractors verbatim and hurt downstream answering.

## History-Length Strategy

Dialogue length is not forced by template. The pipeline first generates an
oversized candidate pool and observes the natural full-history token
distribution.

If the distribution is compact, a common token range can be used as a quality
gate. If it is dispersed, the project keeps the natural variation and stratifies
samples into short / medium / long history-length bins, preferably by quantiles
after quality filtering.

History length is treated as a context-pressure variable. The analysis checks
whether architecture effects change across short, medium, and long dialogue
histories.

## Sampling Plan

Current formal setting:

- 100 samples x 3 conditions = 300 answer-model inferences,
- generated from an oversized formal pool and filtered through the full-history
  gate,
- followed by a 22-case manual annotation of critical compressed-condition
  failures.

Optional follow-up settings:

- budget sweep across smaller/larger compressed-history targets,
- multi-model validation,
- Layer 2 external validation on a separate long-context MC subset.

Formal sampling balances:

- hop count,
- evidence position profile,
- whether answer-critical evidence appears in the recent-turn window,
- full-history token bin,
- MC and dialogue audit quality.

Samples are not selected based on model answer correctness.

## Evaluation

Primary metric:

- final-answer accuracy.

Intermediate compression-quality metrics:

- required evidence retention,
- answerability after compression,
- hallucinated fact count,
- stale or irrelevant information handling,
- compressed-history token count and compression ratio,
- parse success rate.

The key analysis is:

- whether aggregate accuracy hides architecture-by-position interactions,
- whether hybrid recent-turn retention helps when answer-critical evidence is in
  the recent window and hurts when relevance lies in older history,
- whether the position-conditioned effect changes across evidence-position
  profiles and history-length bins,
- which mechanisms explain critical failures after the main accuracy analysis.

Manual mechanism audit is secondary evidence. It explains the main interaction
rather than defining the primary result. The audit uses this schema:

- `compression_effect`: evidence omitted, distractor overweighted,
  relation-structure blurred, exact-value collapse, recent-distractor
  interference, or no major compression loss;
- `downstream_error`: wrong entity/numeric/time-period anchor, exact value lost,
  or recency distractor latch;
- `compression_vs_reasoning`: compression-primary, reasoning-primary, or mixed;
- distractor volume, local competition density, recent verbatim effect, hybrid
  benefit source, answer quality, and audit confidence.
